# Traditional PRD vs. AI-Native PRD: Section-by-Section Comparison

> The table below maps standard PRD sections to their AI-native equivalents. The goal is not to replace the standard PRD — it's to replace the parts that break down when the system is non-deterministic.

| PRD Section | Traditional Approach | AI-Native Approach | Why it matters |
|---|---|---|---|
| **Acceptance criteria** | A list of functional behaviors the feature must exhibit, verified by QA pass/fail testing | Eval criteria with a numeric target, a measurement method, and a golden dataset — evaluated before, during, and after launch | A language model that "works" in QA can still fail in production; you need a measurement system, not a checklist |
| **Error handling** | Defined for known error states: null responses, API timeouts, validation failures | Fallback behavior spec by failure type: API error, low-confidence output, and post-hoc detected wrong output — each with distinct user-facing behavior and a fallback owner | AI systems fail in ways that are probabilistic and often silent; "graceful degradation" is not a design until you've specified what the user sees at each failure type |
| **Performance requirements** | End-to-end response time target, usually a single p95 number | Latency SLA defined per model call in the pipeline — retrieval, reranking, generation — each with p50 and p95, plus a degraded mode trigger | A 3-second pipeline has five places it can break; you can't debug or monitor it without knowing which component owns which part of the budget |
| **Feature toggle / kill switch** | Binary on/off flag to disable the feature in production | Confidence threshold spec defining three action tiers — autonomous, suggest-and-confirm, abstain — plus a kill switch with a defined rollback trigger | Turning off a non-deterministic feature is not enough; you need to define when the model should act vs. defer before it ever ships |
| **Testing plan** | QA test cases covering functional requirements; regression suite for known behaviors | Eval pipeline with a defined golden dataset, an LLM-as-judge or human review cadence, and a regression alert threshold | You cannot write a test case for every possible model output; you need a measurement system that runs continuously, not a suite that runs once |
| **Dependencies** | System dependencies, third-party APIs, infrastructure requirements | Data readiness gates — specific datasets with quality bars, PII handling requirements, and explicit go/no-go criteria — plus model and vendor risk flags | Data problems discovered after build start are expensive; treating data readiness as a gate (not background context) is the difference between a blocked sprint and a blocked launch |
| **Launch criteria** | Feature is complete, QA has signed off, stakeholder demo approved | Pre-mortem completed with mitigations documented, rollout risk flags reviewed, cost-at-scale approved, rollback plan tested | AI features can pass QA and still degrade in production within weeks; launch criteria must include a plan for what happens when the model is wrong, not just when the infrastructure is broken |
| **Post-launch monitoring** | Error rate, uptime, and user engagement metrics tracked in dashboards | Model drift detection via recurring eval runs with regression alerts, plus acceptance rate and feedback signal monitoring | A model that worked at launch can degrade as the world changes; monitoring without eval regression detection will miss the failure until users tell you |

---

*This comparison is intentionally opinionated. Not every AI feature needs every AI-native section at the same depth — but if you're skipping a section, you should be able to explain why it doesn't apply, not just that you ran out of time.*
